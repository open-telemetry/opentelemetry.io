Apache Arrow を用いた OpenTelemetry Protocol（OTel-Arrow）プロジェクトの OpenTelemetry Collector 向け[エクスポーター](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/635d4254a3018eb3ca8f1736e71fcb54f8ed6e5a/exporter/otelarrowexporter?from_branch=main)と[レシーバー](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/635d4254a3018eb3ca8f1736e71fcb54f8ed6e5a/receiver/otelarrowreceiver?from_branch=main)コンポーネントは、現在 OpenTelemetry Collector-Contrib のリリースに含まれています。
これは、ServiceNow Cloud Observability において、社内テレメトリーデータの主要な取り込みパスとして OTel-Arrow コンポーネントを使用した OpenTelemetry Collector のデプロイ経験に関するケーススタディです。

F5, Inc. が OpenTelemetry プロジェクトに最初のコントリビューションを行って以来、F5 を含むコミュニティメンバーは、OTel-Arrow エクスポーターおよびレシーバーコンポーネントを、コストのかかるネットワークリンクを介して大量の OpenTelemetry データを転送するための信頼性が高くパフォーマンスに優れたソリューションに変えるため、協力して取り組んできました。
これらのコンポーネントを社内テレメトリーに使用することで、非圧縮サイズに対して 15 倍から 30 倍の圧縮率（15 分の 1 から 30 分の 1 のサイズ）を観測しています。
OpenTelemetry Protocol（「OTLP」）と Apache Arrow を用いた OpenTelemetry Protocol を同様に構成されたトレースパイプラインで並行比較したところ、圧縮率が 30% 改善されることが確認されました。
この調査は特にトレースデータに焦点を当てていますが、ログやメトリクスシグナルについても本番環境で同様の結果を観測しており、OTel-Arrow ユーザーは同様のパイプライン構成において OTLP と比較して 50% から 70% の改善を期待できます。

これまでの実験結果が本番環境で検証されたことで、Apache Arrow を用いた OpenTelemetry Protocol のエクスポーターおよびレシーバーは一般利用可能な状態にあると考えています。

## エクスポーターとレシーバーのコンポーネント {#exporter-and-receiver-components}

[Apache Arrow](https://arrow.apache.org/) は OpenTelemetry にとって優れた技術的依存先です。
なぜなら、これらのプロジェクトには多くの共通点があるからです。
OpenTelemetry と同様に、Apache Arrow は共有データ仕様、トランスポートプロトコル、クロス言語 API 仕様、そしてコミュニティが構築した多数のランタイム環境向けの SDK を備えています。

Apache Arrow は今やカラム指向データのデータ処理業界における標準であり、相互運用性とパフォーマンスの両面から広く利用されています。
ここでは、このエキサイティングな技術を使用して、OpenTelemetry Collector 間の圧縮ブリッジを構築する方法を説明します。

圧縮ブリッジは、エクスポーターとレシーバーとラベル付けされた 2 つの OpenTelemetry Collector、あるいは 2 つのロードバランスされた Collector プールで構成されます。

[OTEP 0156 設計ドキュメント](https://github.com/open-telemetry/opentelemetry-specification/blob/5239912e97a4df50b206f5fa8241dd6cd8d2dbf8/oteps/0156-columnar-encoding.md?from_branch=main#mapping-otel-entities-to-arrow-records)に記述されているように、エクスポーターは任意の OpenTelemetry データを Arrow レコードバッチに変換します。
Arrow レコードバッチは標準化されたレイアウトを持つメモリブロックであり、データをコピーすることなくアドレス空間や仮想プロセス境界を越えてデータを交換することを可能にします。
この「ゼロコピー」アプローチにより、Arrow は外部関数の呼び出しや、Arrow レコードバッチ上で動作するよう任意の言語でコンパイルされたプロセッサー固有の実行「カーネル」の利用を可能にしています。

標準的な OpenTelemetry Protocol データは、リソース、スコープ、および個々のテレメトリー項目の階層構造として構成されています。
スパン、ログ、メトリクスをエンコードする前に、個別のリソースとスコープの値が識別・重複排除され、OTel-Arrow レコードバッチ内の個別の構造としてエンコードされます。
ここでは、データの局所性を改善するための項目のソートなど、その他の最適化も行います。

Apache Arrow プロジェクトはプロセス間通信（IPC）のためのプロトコルを規定しています。
[Arrow IPC](https://arrow.apache.org/docs/format/DissociatedIPC.html) は「分離された」プロトコルであり、一連の Arrow レコードバッチを送信するための特定のトランスポート方法には縛られていません。
Apache Arrow には独自の意見を持った RPC フレームワーク、Arrow Flight がありますが、私たちの目的のためには、Arrow IPC を gRPC ストリーム内に埋め込むことで、OpenTelemetry Collector の既存の gRPC 設定、サービス、認証エクステンション、および計装を活用することが自然でした。
この設計により、高度に最適化された Arrow IPC 関数を利用して繰り返されるレコードバッチを圧縮・エンコードしつつ、OTLP/gRPC サービス設定とのドロップイン互換性を提供できます。

OTel-Arrow レシーバーはエンコーダーで行われた変換を逆転するロジックを適用し、元のデータと等価なデータが再構築されて、OpenTelemetry Collector パイプラインの次のコンポーネントに供給されます。
将来的には、SDK からプロセッサーまで、OpenTelemetry のエンドツーエンドで Arrow を使用することで多くの利点が得られると予想していますが、そのトピックは今後の投稿に譲ります。

## パフォーマンスとスケーラビリティ {#performance-and-scalability}

ユーザーやサービスプロバイダーは、OTel-Arrow エクスポーターおよびレシーバーコンポーネントを、OpenTelemetry Collector のコア OTLP gRPC エクスポーターおよびレシーバーコンポーネントのドロップイン置き換えとして使用することで恩恵を受けられます。
OTLP/gRPC、大きなバッチサイズ、および Zstd 圧縮を使用して達成できる最良の結果と比較して、30〜70% のネットワーク帯域幅削減が期待できます。

優れた圧縮が利点であり、ネットワークコストがオブザーバビリティの総コストの大きな要因であることは理解していますが、ユーザーはテレメトリー収集インフラストラクチャの信頼性、スケーラビリティ、およびオブザーバビリティも期待しています。

私たちは ServiceNow Cloud Observability の本番環境において、社内テレメトリーを収集する主要なエージェントとして、OTel-Arrow コンポーネントを使用した OpenTelemetry Collector を 1 年以上運用しており、そこで学んだことを OTel-Arrow コンポーネントと OpenTelemetry Collector 自体の改善にフィードバックしてきました。

ここでは、OTel-Arrow コンポーネントを使用した OpenTelemetry Collector のチューニングにより、信頼性が高くスケーラブルなテレメトリー配信を提供する方法を簡潔に説明します。

## OTel-Arrow エクスポーター {#otel-arrow-exporter}

OTel-Arrow エクスポーターとレシーバーのペアは、コア OTLP/gRPC エクスポーターとレシーバーのペアと同等の機能を提供するように設計されており、ユーザーはこれらのコンポーネント間で切り替えられます。
OTel-Arrow コンポーネントは元々これらのコンポーネントから派生したものであり、両方とも統合サービス内で OTel-Arrow と並行して OTLP/gRPC リクエストのサポートを保持しています。

OTel-Arrow エクスポーターコンポーネントは、OpenTelemetry トレース、ログ、またはメトリクスのバッチを伴う同期エクスポート呼び出しから始まる OpenTelemetry データの変換を行います。
エクスポートコンテキストには、期限、トレースコンテキスト、リクエストごとのヘッダーなどの標準的な gRPC メタデータが含まれ、このコンテキストのすべてが OTel-Arrow ストリームを通過します。
リクエストがストリームにディスパッチされた後、呼び出し元はレスポンスまたは期限の到来を待ち、その後リターンします。

各 OTel-Arrow ストリームは、スキーマ、辞書、および将来のストリームリクエストから参照できる関連データを含む内部状態を維持します。
ストリームリクエストは両方のコンポーネントによって順次エンコード・デコードされ、正しい状態の維持を可能にしますが、各ストリームが処理できるデータ量には制限があります。
エクスポーターストリームの数を増やすとスループットは向上しますが、各ストリームにはオーバーヘッドがあるため、一般的に圧縮結果にはストリーム数が少ない方が有利です。

OTel-Arrow ストリームは gRPC ストリームであり、HTTP/2 ストリームにマッピングされます。
ストリームの存続期間は、中間ロードバランサーによってネゴシエーションされた制限を含む多くの要因によって決定されます。
OTel-Arrow エクスポーターは最大ストリーム存続期間の設定をサポートしており、インターバルごとにストリームを自動的に再起動します。
圧縮はストリームの存続期間が長いほど改善されますが、効果は逓減します。

OTel-Arrow エクスポーターは、gRPC ベースのエクスポーター用の Collector 組み込み設定メカニズム（たとえば、エンドポイント、ヘッダー、TLS）を使用し、キュー、リトライ、タイムアウト動作を含む Collector の標準エクスポーターサポートを使用します。
コンポーネントにはエンドポイント設定を含む多くの共通点があるため、ユーザーは既存の OTLP エクスポーターを大幅な再構成なしに OTel-Arrow エクスポーターに置き換えることができます。

設定例を以下に示します。

```yaml
exporters:
  otelarrow:
    endpoint: collector.local:4317
    tls:
      insecure: true
    arrow:
      num_streams: 1
```

## OTel-Arrow レシーバー {#otel-arrow-receiver}

OTel-Arrow レシーバーコンポーネントは、受信した OTel-Arrow ストリームを管理します。
レシーバーは、ストリームごとのリーダーおよびライタースレッドを持つ非同期プロセスを調整します。
パイプライン操作は独立したワーカースレッドを使用して実行され、各ストリームのスループットを最大化できます。
レシーバーはいくつかの制限の下で、可能な限り多くの並行作業を処理します。
OTel-Arrow レシーバーのメモリ使用量を制御する 2 つの主要な設定パラメーターがあります。

1 つ目の制限は、スキーマ、辞書、および関連データを含むアクティブストリームが使用するメモリの合計を管理します。
メモリ制限に達すると、レシーバーはリソース枯渇ステータスコードでストリームを終了します。
これにより、予想外に大きなストリームメモリ要件によってレシーバーがメモリ不足になることが防止されます。

2 つ目の制限は、パイプラインに入るデータ量を管理し、OTLP と OTel-Arrow の両方のデータパスをカバーします。
この制限に達すると、個別のリクエストまたはストリームはブロックされるか、同時待機数の制限によって即座に失敗します。
アドミッション制限は、パイプラインのストールによるレシーバーのメモリ不足を防止します。

OTel-Arrow パイプラインでのメモリ制御にはこれらの制限の使用を推奨し、標準の memory-limiter プロセッサーの使用は推奨しません。
メモリとロードバランシングに役立つ、gRPC メッセージサイズ制限、ストリームごとの接続制限、キープアライブ制限などの標準 gRPC 設定を通じて利用可能なその他の制限もあります。

個々のレシーバーの負荷が大きくなりすぎた場合は、接続数、ストリーム数、および接続の存続期間に制限を設けるために、外部の HTTP/2 ロードバランサーの使用を推奨します。

設定例を以下に示します。

```yaml
receivers:
  otelarrow:
    protocols:
      grpc:
        max_recv_msg_size_mib: 16
      arrow:
        memory_limit_mib: 128
    admission:
      request_limit_mib: 128
```

## OTel-Arrow ストリームにおけるバッチ処理とバックプレッシャー {#batching-and-back-pressure-in-otel-arrow-streams}

バッチサイズは圧縮パフォーマンスに大きな影響を与えます。
一般的にバッチが大きいほど圧縮率は向上しますが、コストも伴います。
大きなバッチはより多くのメモリを必要とし、CPU 使用量が増え、パイプラインにネットワークおよび CPU レイテンシーを追加するため、これらの要因をバランスさせる必要があります。
OpenTelemetry Collector ユーザーには、並行性、キューイング、永続性を含む他の要因に関連した、いくつかのバッチ処理オプションが利用可能です。

他のソースからデータを受信し、OTel-Arrow エクスポーターを介してエクスポートするゲートウェイ Collector には、完全に同期的なリクエストパスを推奨します。
呼び出し元は、元のデータを解放する前にパイプラインがリクエストを確認するのを待ちます。
この方法では、個々のストリームが飽和すると、パイプラインがより多くのデータを処理するにつれてレイテンシーとメモリ使用量が上昇します。
この応答は「バックプレッシャー」と呼ばれ、物理的なパイプラインにおける圧力に類似しています。
バックプレッシャーにより、テレメトリープロデューサーはパイプラインに問題があることを知り、別の接続でリトライする機会を得られます。
また、サービスオペレーターはメモリ使用率に基づいてインスタンス数をオートスケールし、負荷の増加に対処する機会を得られます。

詳細については、OTel-Arrow エクスポーターの[バッチ処理とバックプレッシャーに関するドキュメント](https://github.com/open-telemetry/opentelemetry-collector-contrib/blob/f4661d486acbbef5c4fb071adafe5818035d2512/exporter/otelarrowexporter/README.md?from_branch=main#batching-configuration)をご覧ください。

## パフォーマンス {#performance}

ServiceNow Cloud Observability では、社内トレースおよびログデータのゲートウェイとして動作する OpenTelemetry Collector を評価しました。
サンプリング後に平均 50 万〜60 万スパン/秒を OTel-Arrow ブリッジ経由で書き込み、250〜300 MB/秒の範囲の圧縮データをエクスポートする社内トレースパイプラインを使用して、一連の実験を実施しました。
OTel-Arrow エクスポーターとレシーバーの組み込み OpenTelemetry 計装を使用して、圧縮、失敗率、中央値レイテンシーの観点からパイプラインのパフォーマンスを定量化しました。

実験のセットアップを以下の図に示します。
ここでは、データが左から右に流れる様子が描かれており、OpenTelemetry で計装されたアプリケーションが、ラウンドロビンロードバランシングを使用した標準の OTLP over gRPC を使用するゲートウェイ Collector のデプロイメントにテレメトリーを生成しています。
ゲートウェイ Collector は、上述の concurrent batch プロセッサーを含むさまざまなプロセッサーを適用した後、OTel-Arrow エクスポーターへと続きます。

![ゲートウェイ Collector のプールがロードバランサーに送信し、その後バックエンドサービスに送信します。](setup.png)

ブリッジの反対側では、Envoy ロードバランサーのプールが TLS を終端した後、OTel-Arrow レシーバーを実行する Collector のプールにストリームを分散します。
Envoy にはラウンドロビン設定を使用しています。
これは、長寿命のストリームのバランシングにおいて、最小負荷ポリシーよりも良好なパフォーマンスを示すことがわかったためです。
負荷バランスを改善するために、Envoy と OTel-Arrow レシーバーの HTTP/2 max_concurrent_streams 設定の両方を 1 に設定しています。

パイプラインは完全に同期的であり、元の OTel SDK とすべての中間パイプラインステージがバックエンドがすべてのリクエストに応答するのを待ちます。
どちらの Collector プールもリトライするようには設定されていません。
かわりに、OpenTelemetry SDK がリトライするように設定しています。

### 実験方法 {#experimental-method}

これらの実験は稼働中の本番システムで実施されたものであり、OTel-Arrow ブリッジは他の設定済みプロセッサーやプロプライエタリバックエンドなどの変数を含むエンドツーエンドパイプラインの一部であるため、これらの結果はバルクテレメトリー転送における圧縮パフォーマンスとコスト面での優位性の相対的な改善を示すことのみを意図しています。

以下の実験結果における各トライアルは 2 時間の長さで、測定開始前にテスト対象構成で少なくとも 1 時間のランタイムを経ています。
トライアルが有効とみなされるためには、平均成功率が 99.95% を超え、中央値パイプラインレイテンシーが 3 秒未満でなければなりません。
テストされた構成では、ゲートウェイ Collector インスタンスのプールが約 35,000 スパン/秒を受信し、サンプリング後に vCPU/秒あたり約 7,000 スパン/秒をエクスポートします。
ゲートウェイ Collector は、CPU 使用率 60% を目標とする Kubernetes の水平ポッドオートスケーリングポリシーによって管理されています。

OTel-Arrow コンポーネントは、OTLP と OTel-Arrow の両方のプロトコルについて、otelcol_exporter_sent、otelcol_exporter_sent_wire、otelcol_receiver_recv、および otelcol_receiver_recv_wire メトリクスを使用して、圧縮済みおよび非圧縮のバイト数を一貫して報告します。
実験結果は、ServiceNow Cloud Observability のメトリクスクエリを使用した時系列クエリの結果として報告されます。
これらのメトリクスは gRPC メソッド名で便利にタグ付けされているため、本番環境での圧縮パフォーマンスのモニタリングが容易です。

例として、エクスポーターの観測圧縮率は以下のクエリを使用して各エクスポートメソッドについて算出されます。

```text
with
    uncompressed = metric otelcol_exporter_sent
        | rate
        | group_by ["method"], sum;
    compressed = metric otelcol_exporter_sent_wire
        | rate
        | group_by ["method"], sum;
join uncompressed / compressed
```

圧縮の数値は圧縮比ではなく削減率で報告しているため、値が大きいほどパフォーマンスが向上していることを意味します。
圧縮比は削減率の逆数として計算できます。
たとえば、削減率 20 は結果が 20 分の 1 のサイズになることを意味し、圧縮比 5% に相当します。

OpenTelemetry Collector および OTel-Arrow コンポーネントのバージョン 0.105.0 がテストされました。

### 実験：ストリーム存続期間の関数としての圧縮 {#experiment-compression-as-a-function-of-stream-lifetime}

この実験では、OTel-Arrow エクスポーターの最大ストリーム存続期間を変化させました。
仮説は、初期の調査結果で報告したように、ストリームが長いほどデータ量が多くなり、圧縮の機会が増えるため、圧縮が改善されるというものです。
この実験では、4000〜5000 スパンのバッチが 250〜300KiB の範囲のサイズに圧縮されます。

3.75 秒から 4 分までの 5 つの最大ストリーム存続期間の値がテストされ、ストリーム存続期間と圧縮の間に期待される関係が確認されました。
4 分を超えるストリーム存続期間はテストされていません。
これは単に、テスト対象の OTel-Arrow レシーバーがストリームの存続期間を 5 分に制限する gRPC キープアライブ設定を持っているためです。

| ストリーム存続期間 | エクスポーター削減率 |
| ------------------ | -------------------- |
| 3.75s              | 15.6                 |
| 15s                | 16.3                 |
| 1m                 | 16.9                 |
| 2m                 | 17.2                 |
| 4m                 | 17.1                 |

OTLP プロトコルを通過する同様のデータの平均削減率は 12.0 です。

OTel-Arrow エクスポーターが観測する圧縮削減率は、通常レシーバーが観測するものよりも 1% 高くなります。
これは、両者が同じ圧縮データを観測する一方で、OTLP のリソースおよびスコープ値の階層構造内での重複排除により、ブリッジの各側で非圧縮サイズが異なるためです。
たとえば、1 分の存続期間テストでエクスポーターがデータを 16.9 倍に削減した場合、レシーバーは非圧縮サイズの変化により 16.8 倍にしかデータを展開しませんでした。

### 実験：バッチサイズの関数としての圧縮 {#experiment-compression-as-a-function-of-batch-size}

別の実験では、上述の concurrent batch プロセッサーで使用されるバッチサイズパラメーターを変化させました。
バッチサイズが大きいほど圧縮が改善されることはよく知られていますが、同時にメモリ使用量とパイプラインレイテンシーも増加します。
通常、バックエンドサービスにはリクエストサイズの上限があり、バッチサイズを増やすことの利点に上限が設けられます。

OTel-Arrow ブリッジは、同様に構成された OTLP トレースパイプラインと比較されました。
Apache Arrow を用いた OpenTelemetry Protocol は、ネットワーク帯域幅を約 30% 改善することが確認されました。

| バッチサイズパラメーター（最小、最大スパン数） | OTel-Arrow エクスポーター平均削減率 | OTLP エクスポーター平均削減率 |
| ---------------------------------------------- | ----------------------------------- | ----------------------------- |
| 1000–1250                                      | 16.4                                | 11.9                          |
| 2000–2500                                      | 17.2                                | 12.0                          |
| 4000–5000                                      | 17.7                                | 12.2                          |

### 実験：OTLP との差分コスト比較 {#experiment-differential-cost-comparison-with-otlp}

プロジェクト開始時の仮説の 1 つは、バルクテレメトリーデータを送信するユーザーにとって、圧縮の利点がエンコーディングに要する追加のコンピュートリソースのコストを上回るというものでした。
私たちはブリッジのエクスポーター側のこのメトリクスに特に関心を持っています。
なぜなら、テレメトリーをサービスプロバイダーに送信するユーザーのオブザーバビリティ総コストに影響するからです。
ここでは、標準 OTLP データの送信コストと OTel-Arrow データの送信コストを比較します。

OTel-Arrow エクスポーターは対応する OTLP エクスポーターよりもやや多くのメモリを使用しますが、その差はコストの大きな要因にはなりません。
これは、ガベージコレクションなどの他の要因により、2 つのエクスポーターが同様のメモリ量を必要とすること、そしてテストされたマシンサイズではメモリコストが CPU コストの 10% であることが理由です。

私たちは、2 つのエクスポーター構成で費やされたコストと vCPU/時間の数、および送信されたテレメトリーの GiB/時間の数の比較に関心を持っています。
クラウドコンピューティング環境では、1 vCPU/時間の価格がパブリックインターネットへの 1 GiB のエグレスコストよりも低いことが一般的です。
通常、1 vCPU/時間のコストは 1 GiB のエグレスコストの 4 分の 1 から 2 分の 1 程度と推定されます。

このロジックに従うと、vCPU/時間の差とエクスポートされた GiB の差を比較することで、OTel-Arrow ブリッジが経済的に理にかなっていることがわかります。
たとえば、あるトライアルでは、OTel-Arrow エクスポーターは平均 77.0 vCPU を使用し、107 GiB/時間の速度でエクスポートしました。
同様の量の集約データを使用した同様のトライアルでは、OTLP エクスポーターが 53.7 vCPU を使用し、143 GiB/時間をエクスポートしたことが観測されました。
ほとんどのクラウドコンピューティング契約において、23 vCPU/時間のコストは 36 GiB/時間のコストよりも大幅に低いため、Apache Arrow を用いた OpenTelemetry ブリッジを採用することでテレメトリー転送の総コストが削減されます。

## まとめ {#summary}

私たちは、さまざまな OTel-Arrow エクスポーター構成のパフォーマンスを報告しました。
いずれのケースにおいても、信頼性やスケーラビリティを犠牲にすることなく、OpenTelemetry OTLP エンコーディングと比較して大幅な圧縮の利点を提供しています。
ServiceNow Cloud Observability の社内トレースおよびログデータを使用して、OTel-Arrow は非圧縮データの 15 分の 1 から 30 分の 1 の圧縮削減を達成し、一般的に OTLP プロトコルよりも 30% から 50% 少ないネットワーク帯域幅を実現しています。

これは Apache Arrow を用いた OpenTelemetry Protocol プロジェクトの始まりにすぎません。
Apache Arrow レコードバッチ処理の利点を、OpenTelemetry Collector パイプラインのさらに奥へと拡張することを期待しています。
OpenTelemetry SDK がこのフォーマットを直接生成し、テレメトリープロセッサーが Apache Arrow ライブラリに基づいて記述できる、大幅に低い全体コストを持つエンドツーエンドの Apache Arrow を用いた OpenTelemetry Protocol パイプラインを楽しみにしています。

さらに、OTel-Arrow ブリッジが今日のテレメトリー全体のコストを削減できることを示しました。
Apache Arrow を用いた OpenTelemetry Protocol は本番環境での使用に対応しています。
ユーザーとテレメトリーシステムに対し、低コストなテレメトリー転送のために Apache Arrow を用いた OpenTelemetry Protocol コンポーネントの採用を呼びかけます。
これに興味を持ち、OpenTelemetry と Apache Arrow のさらなる統合を見たい場合は、ぜひコミュニティに参加するか、[プロジェクトリポジトリ](https://github.com/open-telemetry/otel-arrow)にイシューを投稿してください。
